# API Service Builder

## 执行目标
- 统一请求风格，复用现有 `src/utils/request.js` 拦截器能力。
- 让页面与请求层解耦，提升复用和可维护性。
- 提供清晰的入参与返回约定，减少页面层样板代码。

## 执行步骤
1. 先确认接口所属业务域，再创建或更新 `src/api/<domain>.js` 模块。
2. 在 API 模块中封装具名函数，返回 `request({...})` 调用结果。
3. 查询接口使用 `params`，新增/编辑接口使用 `data`，语义化命名方法。
4. 在页面或 store 中按业务流程调用 API 函数，处理加载状态与结果回填。
5. 避免在调用层重复处理全局 401 跳转和通用报错提示。

## 项目约定
- 基础实例来自 `src/utils/request.js`，不要重复创建新的 axios 实例。
- 鉴权 Token 由请求拦截器注入，不要在每个 API 手工拼接请求头。
- 业务特定错误可在调用层补充提示；通用错误交给全局拦截器。
- API 文件按业务拆分，避免把全部接口堆在一个文件中。

## 完成检查
- 检查每个接口方法都具备明确入参和返回使用方式。
- 检查页面请求流程含加载态、成功反馈、失败回退。
- 检查接口路径、方法（GET/POST/PUT/DELETE）与业务一致。
- 检查未新增重复 axios 配置或与拦截器冲突的逻辑。
